# 설명 (STAR)

## Situation
- 목표: 기업 관련 뉴스/재무 데이터를 모아 간단한 투자 리포트를 자동으로 보여주는 데모 앱 필요.
- 제약: 빠르게 동작하는 데모 수준, 로컬 MySQL 환경 사용, 복잡한 LLM API 의존 최소화.

## Task
- Streamlit UI에서 입력값으로 뉴스/재무 수집 → 임베딩 기반 검색 → 요약 리포트까지 한 번에 수행.
- MySQL에 임베딩을 저장하고, 유사도 검색이 가능하도록 구현.
- 실행 문서화와 설정(.env) 제공.

## Action
- UI/흐름: `app.py`에서 데이터 수집 → 임베딩 생성 → 저장 → 검색 → MoE 요약 순서로 연결.
- 뉴스/재무 수집: Google News RSS와 yfinance로 빠르게 결과를 확보하도록 구현.
- 임베딩: `sentence-transformers/all-MiniLM-L6-v2`를 기본으로 사용해 로컬에서도 실행 가능한 경량 모델 선택.
- DB: MySQL을 사용하되, 벡터 전용 확장이 없으므로 임베딩을 JSON 문자열로 저장하고 앱에서 코사인 유사도로 계산.
- MoE: 복잡한 라우팅 대신 요약/감성/재무/리스크 섹션으로 단순화해 데모에 필요한 가독성 확보.

## Result
- 최소 설치로 실행 가능한 데모 앱 완성.
- 입력값만으로 뉴스/재무/요약 리포트 출력 가능.
- MySQL 환경에서도 동작하도록 임베딩 저장/검색 파이프라인을 단순화.

## 선택 이유와 대안
- MySQL + JSON 임베딩 저장 선택 이유
  - 데모 목적에서 운영 환경 구축 복잡도를 낮추기 위해 MySQL 그대로 활용.
  - 대안: PostgreSQL + pgvector 사용 시 검색 성능/정확도는 좋아지지만 설치 난이도와 운영 복잡도가 증가.
- `sentence-transformers` 사용 이유
  - 로컬 실행이 가능하고 설치가 단순하며, 384 차원으로 속도/성능 균형이 좋음.
  - 대안: OpenAI/Gemini 임베딩 사용 시 품질은 상승 가능하지만 비용/키 관리가 필요.
- MoE를 규칙 기반으로 단순화한 이유
  - 데모에서는 구조 확인이 우선이라 복잡한 LLM 라우팅보다 설명 가능한 요약이 적합.
  - 대안: LLM 기반 라우팅/요약은 품질이 좋지만 비용 및 안정성 이슈가 존재.
